# 忆织知识提取

## 触发场景

当用户在对话中**主动要求**以下操作时使用此技能：
- "帮我提取这段对话的知识"
- "总结一下刚才讨论的要点"
- "提炼一下这段排查的思路"
- "复盘一下刚才的选型过程"

## 任务目标

你是一个严格结构化的数据提取程序。请严格按照以下过滤规则和分类逻辑，分析当前对话记录并提取核心信息。

## 步骤一：基础过滤（强制执行）

回顾当前对话记录，根据以下两方面原则进行过滤：

**原则一：通用规则（以下内容不提取）：**
- 超过 5 行的具体代码实现或长篇系统报错日志
- 文本润色、翻译、日常闲聊
- 常规的软件安装步骤或环境依赖排查
- 简单的语法错误纠正、单词拼写、格式排版
- 基础的工具使用说明、API 参数查询（搜索引擎能快速查到的内容）
- 没有逻辑推导过程的机械问答

**原则二：复利规则（读取用户的历史偏好）：**
请读取文件 `~/.memloom/feedback/rejected.md`（如果存在），其中记录了用户曾经删除过的知识类型。这些内容也不应再次提取。

**如果根据以上两方面原则过滤后，对话中没有剩余有价值的内容**，直接告知用户「📭 当前对话暂无值得提取的核心知识」并停止。否则继续。

## 步骤二：场景分类与特定提取规则

对通过过滤的有价值内容，判断其所属场景，按对应规则提取：

1. **编程与架构场景：**
   - 必须提取：系统选型时的妥协与取舍（牺牲了什么换取了什么）、导致复杂问题的底层运行机制（如内存分配、并发冲突机制）
   - 禁止提取：常规的业务增删改查逻辑、特定框架的基础用法、正则表达式写法

2. **论文与学术场景：**
   - 必须提取：该研究填补的核心理论空白、推导结论所依赖的关键假设或边界条件
   - 禁止提取：论文摘要的直白翻译、基础数学公式的推算过程、常规的文献列举

3. **其他通用场景：**
   - 必须提取：解决某一类复杂问题的通用思维模型或因果分析逻辑

### 知识粒度原则（严格遵守）

**一次连贯的问答 = 一个知识单元 = 一张卡片。** 判断粒度的依据是对话结构，而不是内容中涉及了几个子概念：

- 如果用户问了一个问题，AI 给了一个完整回答（即使回答中包含多个对比项、多个子概念），这属于**一个知识单元**，应当合并为一张卡片
- 例如：用户问"同步和异步共识的区别"，AI 回答中分别介绍了两者 → 这是**一张卡片**（标题如"同步与异步共识的核心权衡"），不要拆成"同步共识"和"异步共识"两张
- detail 中应当覆盖子概念间的**关联、对比和权衡**，而不是孤立地描述每个子概念
- **只有当对话中出现了明确不同的话题转换**（用户问了完全不相关的新问题），才应生成多张卡片

## 步骤三：去重检查（必须执行）

在生成卡片之前，你必须对每个候选知识点执行以下去重流程：

1. **索引扫描**：阅读系统注入的「忆织知识索引」表，检查是否存在标题或核心概念高度相似的已有条目
2. **API 确认**：如果在索引中发现可能重复的条目，执行以下命令搜索已有卡片：
   ```bash
   curl -s 'http://127.0.0.1:3000/api/cards?keyword=<关键词>'
   ```
3. **比对判定**：对比候选知识与已有条目的 detail，如果核心概念和阐述逻辑基本一致（即使措辞不同），则判定为重复
4. **跳过重复**：确认重复的知识点不生成卡片，在最终结果中告知用户「已跳过 N 条与已有知识重复的内容」

> 仅当索引中完全没有相似条目时，可跳过步骤 2-3 直接进入步骤四。

## 步骤四：构造卡片并校验

对每个候选知识点，构造以下字段：

- **title**: 15 字以内，提炼核心概念
- **category**: 从系统注入的当前分类列表中选择
- **tags**: 相关标签数组
- **brief**: 100 字以内，高度概括其运作机制
- **detail**: 用 150-300 字，客观严密地阐述其底层逻辑和成立条件。禁止包含原对话中的大段代码或公式，必须转化为通用的原理解释。如果知识点本身是事实性声明（如政策定义、历史事件），不要仅罗列事实，而应侧重阐明它为什么被提出、它解决了什么底层问题、以及它在技术或理论体系中的结构性位置。
- **review_question**: 构造一道必须运用 detail 中原理进行推理或取舍分析才能回答的场景应用题。规则：1. 必须设定一个 detail 中未直接出现的全新具体场景；2. 答案必须依赖 detail 中的原理进行推导，不能靠背诵事实回答；3. 自检：如果此题可以仅靠记住一个名称、日期、定义就答对，必须重新出题。

构造完成后，自行校验所有字段是否满足以上要求（title ≤ 15 字、brief ≤ 100 字、detail 150-300 字、各字段非空）。如有问题，立即修正。

## 步骤五：执行入库

校验通过后，将卡片通过 HTTP API 入库。执行以下命令（可一次提交多张卡片）：

```bash
curl -s -X POST http://127.0.0.1:3000/api/cards/ingest \
  -H 'Content-Type: application/json' \
  -d '{"cards": [{"title": "...", "category": "...", "tags": [...], "brief": "...", "detail": "...", "review_question": "..."}]}'
```

成功后 API 返回 `{"ingested": N, "cards": [...]}` 。向用户展示成功入库的结果。
