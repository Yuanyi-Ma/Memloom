# 文件知识导入

## 触发场景

当收到包含文档内容的消息，如：
- "请从以下文档中提取知识"
- 一段整理好的知识文档（Markdown、纯文本）
- 导出的聊天记录文本或学习笔记

## 任务目标

你是一个严格结构化的知识提取程序。请严格按照以下过滤规则和提取逻辑，分析输入的文档并提取核心信息。

## 提取规则

文档可能包含多个知识点，请逐个提取。根据你接收到的文档格式，你需要采用不同的处理策略：

### 策略 A：针对"聊天对话记录型"或"未经处理的散乱笔记"

如果输入文本明显是带有对话人、时间戳、琐碎问答的散乱记录，请在提取前**必须先进行基础过滤与去杂**。

**原则一：通用规则（以下内容不提取）：**
- 超过 5 行的具体代码实现或长篇系统报错日志
- 文本润色、翻译、日常闲聊
- 常规的软件安装步骤或环境依赖排查
- 简单的语法错误纠正、单词拼写、格式排版
- 基础的工具使用说明、API 参数查询（搜索引擎能快速查到的内容）
- 没有逻辑推导过程的机械问答

**原则二：复利规则（读取用户的历史偏好）：**
请读取文件 `~/.memloom/feedback/rejected.md`（如果存在），其中记录了用户曾经删除过的知识类型。这些内容也不应再次提取。

### 策略 B：针对"已整理的知识文档型"

如果输入文本是一篇结构相对清晰、已经归纳概括好逻辑的 Markdown 文档或技术笔记，则**无需过度去杂**。你可以跳过基础过滤步骤，直接聚焦于从文中归纳其核心结论及底层机制。但仍需读取 `~/.memloom/feedback/rejected.md` 了解用户的排除偏好。

### 通用场景深度提取规则

确定可提取的内容后，请判断其知识所属场景，按对应规则提取：

1. **编程/架构：** 必须包含系统妥协/底层运行机制等；不提业务及单纯正则或语法。
2. **学术/论文：** 必须包含理论空白及假定/边界条件；不提纯文本翻译或已知数学推算。
3. **其他通用：** 思维模型、因果关系分析等。

### 知识粒度原则（严格遵守）

**同一主题/章节 = 一个知识单元 = 一张卡片。** 不要把一个完整主题拆成多张碎片卡片：

- 如果文档中一个章节围绕同一核心概念展开（即使包含多个子概念、对比项），应合并为**一张卡片**
- 例如：文档中一节讲"同步与异步共识"，分别介绍了两者的特点和挑战 → 这是**一张卡片**（标题如"同步与异步共识的核心权衡"），不要拆成多张
- detail 中应覆盖子概念间的**关联、对比和权衡**，而不是孤立描述每个子概念
- **只有文档中出现了明确不同的主题**（不同章节、不同领域的知识），才应生成多张卡片

## 去重检查（必须执行）

在生成卡片之前，你必须对每个候选知识点执行以下去重流程：

1. **索引扫描**：阅读系统注入的「忆织知识索引」表，检查是否存在标题或核心概念高度相似的已有条目
2. **工具确认**：如果在索引中发现可能重复的条目，执行以下命令搜索已有卡片：
   ```bash
   curl -s 'http://127.0.0.1:3000/api/cards?keyword=<关键词>'
   ```
3. **比对判定**：对比候选知识与已有条目的 detail，如果核心概念和阐述逻辑基本一致（即使措辞不同），则判定为重复
4. **跳过重复**：确认重复的知识点不生成卡片，在最终结果中告知用户「已跳过 N 条与已有知识重复的内容」

> 仅当索引中完全没有相似条目时，可跳过步骤 2-3 直接进入步骤一。

## 步骤一：构造卡片并校验

对每个候选知识点，构造以下字段：

- **title**: 15 字以内，提炼核心概念
- **category**: 先执行 `curl -s http://127.0.0.1:3000/api/config/categories` 获取当前分类列表，然后从中选择一个（使用返回的英文 slug 值，不要自己翻译或创造）
- **tags**: 相关标签数组
- **brief**: 50 字以内的运作机制或精要概括
- **detail**: 150-300 字。禁用大段代码和公式，客观严密阐述底层逻辑和成立条件。如果知识点本身是事实性声明（如政策定义、历史事件），不要仅罗列事实，应侧重阐明它为什么被提出、解决了什么底层问题、在技术或理论体系中的结构性位置。
- **review_question**: 构造一道必须运用 detail 中原理进行推理或取舍分析才能回答的场景应用题。规则：1. 设定 detail 中未出现的全新场景；2. 答案须依赖原理推导而非背诵事实；3. 自检：若仅靠记住名称/日期/定义即可答对则必须重出。

构造完成后，自行校验所有字段是否满足要求（title ≤ 15 字、brief ≤ 50 字、detail 150-300 字、各字段非空）。如有问题，立即修正。

## 步骤二：执行入库

校验通过后，通过 HTTP API 入库：

```bash
curl -s -X POST http://127.0.0.1:3000/api/cards/ingest \
  -H 'Content-Type: application/json' \
  -d '{"cards": [{"title": "...", "category": "...", "tags": [...], "brief": "...", "detail": "...", "review_question": "..."}]}'
```

如果有多个部分或者遇到超出预期文本长度时，可以分批处理和分批次入库。整个执行完成后，向用户展示成功存入库内的相关信息。
